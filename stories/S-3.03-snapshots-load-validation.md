# S-3.03: Append-Only Snapshots & Load Validation

## Narrative
- **As a** user with two years of history, **I want** snapshots that can never be rewritten and loads that salvage everything salvageable, **so that** my trend is trustworthy even after a damaged file.

## Acceptance Criteria

### AC-001 (traces to BC-2.02.003 postconditions 1–4, invariants 1–3)
- `take_snapshot` appends with gapless monotonic seq (VP-015), deep-frozen answers (later edits don't alter it — VP-014 byte-stability of past records), catalog pin (id/version/digest), write-time-recomputed cached Score (DI-010), informational `taken_at` (seq authoritative); zero-answer snapshot warns (DEC-004); no mutation/delete API exists at any layer.
- **Test:** `test_BC_2_02_003_append_only()` + proptest VP-014/VP-015

### AC-002 (traces to BC-2.02.004 postconditions 1–2)
- Load validates schema + integrity; first invalid record reported as E-STORE-002 (file, record); valid prefix loads read-only; mutating commands refused on damaged store; tail truncation distinguished from mid-corruption.
- **Test:** `test_BC_2_02_004_salvage()` + fuzz `fuzz_store_parse` (VP-016: any single-byte corruption → correct load or named damage, never silent wrong data)

### AC-003 (traces to BC-2.02.004 postcondition 3, EC-004)
- Hand-edited valid stores load normally; `verify` recomputes all cached Scores (E-STORE-003 on mismatch with engine-version context — VP-017); orphan answers quarantined per E-STORE-005, never entering the scoring answer set, listed by status/verify.
- **Test:** `test_BC_2_02_004_verify_and_orphans()`

### AC-004 (traces to BC-2.02.004 EC-002/EC-003)
- Newer-schema store → E-STORE-004, no load attempt; older-schema → transparent in-memory upgrade, on-disk only via `upgrade` (S-3.05).
- **Test:** `test_BC_2_02_004_schema_versions()`
