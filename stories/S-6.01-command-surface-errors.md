# S-6.01: Command Surface, Exit Codes & Error Rendering

## Narrative
- **As a** user and their scripts, **I want** 11 stable commands with contract exit codes and self-service errors, **so that** both humans and CI can rely on the surface.

## Acceptance Criteria

### AC-001 (traces to BC-6.06.001 postconditions 1–3)
- All 11 commands (init, assess, answer, override, snapshot, status, diff, report, verify, migrate, upgrade) wired to their crates; exit classes 0–8 exactly per taxonomy (VP-080 CI matrix covers each class); verify exception: exit 0 only when ok, 4 on findings; --help per command with example; --version with semver + bundled catalog + engine version + target triple; unknown command suggests nearest.
- **Test:** integration matrix `test_BC_6_06_001_commands_exit_codes()`

### AC-002 (traces to BC-6.06.001 invariant 3, VP-081)
- Diagnostics → stderr, data → stdout, all commands (single documented --json exception lands in S-6.02); clap-generated help golden-tested against interface-definitions.md (ADR-006 drift gate).
- **Test:** `test_BC_6_06_001_stream_separation()` + help goldens

### AC-003 (traces to BC-6.06.002 postconditions 1–3, VP-082)
- Every taxonomy code renders `error[E-XXX-NNN]: cause` + `--> element` + `help: remedy` (non-empty, CI-exhaustive — VP-082); multi-defect itemization; internal errors apologize, assert store-safe, request bug report, no backtrace without --debug.
- **Test:** `test_BC_6_06_002_error_rendering_exhaustive()`

### AC-004 (traces to BC-6.06.001 EC-001..003, BC-6.06.002 EC-002)
- "statu" suggests status (exit 2); foreign flags explained; usage errors precede store access; engine panic caught at boundary → formatted exit-1.
- **Test:** `test_BC_6_06_001_edge_cases()`
