# S-5.01: Status View & the Disclosure Chain

## Narrative
- **As a** returning user, **I want** one honest at-a-glance screen, **so that** I know my band, what moved, what's unknown, and what's capped — with nothing suppressible.

## Acceptance Criteria

### AC-001 (traces to BC-5.05.001 postconditions 1–2)
- Always shown: band (+ "capped from X — floor rule" with gaps when triggered), percent, assessed/total, catalog version, override count; ≥1 snapshot → delta vs latest ("uncommitted changes"); ≥2 → seq-ordered sparkline with series-break discontinuities; quarantined orphans listed.
- **Test:** snapshot tests `test_BC_5_05_001_status_content()`

### AC-002 (traces to BC-5.05.001 postconditions 3–4, DEC-001/DEC-004)
- Baseline-only: "baseline established <date>", no fabricated trend; empty store: "0 of 34 assessed — run `otposture assess`" onboarding tone with honest 0%.
- **Test:** `test_BC_5_05_001_zero_and_baseline_states()`

### AC-003 (traces to BC-5.05.001 invariants 1–3, VP-070/071)
- Shared `Disclosures` struct (floor caps, overrides + rationales, breaks, completeness caveat) consumed here and by every later surface; no suppression path (VP-070 snapshot matrix); status takes no lock, performs zero writes (VP-071 trace check); no personal data (DI-013).
- **Test:** `test_disclosures_struct_complete()` + CI write-syscall assertion

### AC-004 (traces to BC-5.05.001 EC-002..004)
- Override-driven band change annotated; NO_COLOR/narrow/no-TTY degrade gracefully; band-vs-percent divergence explained (DEC-010).
- **Test:** snapshot matrix across terminal modes
