# S-6.02: JSON Output Mode

## Narrative
- **As a** user wiring posture into CI or a dashboard, **I want** stable versioned JSON twins of status/diff/verify, **so that** machines consume the same truth humans see.

## Acceptance Criteria

### AC-001 (traces to BC-6.06.004 postconditions 1–2)
- `--json` on status/diff: exactly one JSON document on stdout (no banners), schema_version'd, full data of the human view (score with band/uncapped/floor/gaps, sub-scores, unknown counts, deltas with full-precision + exact-rational driver points, overrides, breaks); `verify --json` emits the distinct `kind: "verify-results"` document.
- **Test:** `test_BC_6_06_004_documents()` + JSON Schema validation in CI (VP-085)

### AC-002 (traces to BC-6.06.004 invariant 1, VP-086)
- JSON and human outputs project the same computed structures — shared-source goldens compare values across both for identical snapshots (incl. the interface-definitions machine-verified example as a golden).
- **Test:** `test_BC_6_06_004_shared_source_goldens()`

### AC-003 (traces to BC-6.06.004 postconditions 3–4, EC-003)
- Full-precision numbers + `*_exact` rational strings; undefined percent → `"percent": null, "scoreable": false` (never NaN); no personal-data fields (DI-013).
- **Test:** `test_BC_6_06_004_precision_null_handling()`

### AC-004 (traces to BC-6.06.004 EC-001)
- Errors under --json: single JSON error object on stdout `{"error":{code,message,help}}`, human diagnostics suppressed, exit codes unchanged (the documented BC-6.06.001 exception).
- **Test:** `test_BC_6_06_004_json_errors()`
