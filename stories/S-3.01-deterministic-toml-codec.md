# S-3.01: Deterministic TOML Codec

## Narrative
- **As a** user keeping the store in git, **I want** byte-stable serialization where one change = one minimal diff, **so that** posture history is reviewable like code.

## Acceptance Criteria

### AC-001 (traces to BC-2.02.008 postcondition 1, VP-022)
- `emit(&StoreState) -> String` is deterministic: stable key order, fixed datetime/number formatting (rationals as "n/d"), LF endings; identical state ⇒ identical bytes on every platform.
- **Test:** proptest `prop_BC_2_02_008_emit_deterministic()` + cross-platform CI golden

### AC-002 (traces to BC-2.02.008 postcondition 2, VP-024)
- Changing one Answer produces a textual diff confined to that answer's block (+ at most one summary line).
- **Test:** proptest `prop_BC_2_02_008_minimal_diff()`

### AC-003 (traces to BC-2.02.008 postcondition 3, VP-023)
- `parse(serialize(s)) == s` for all valid states; files carry header comments (schema version, in-use warning, DI-013 non-PII note); parse with standard toml crate succeeds.
- **Test:** proptest `prop_BC_2_02_008_roundtrip()` + `test_header_comments()`

### AC-004 (traces to BC-2.02.008 EC-001..003)
- Special characters in notes escape and round-trip; Windows/Linux emit byte-identical; multi-KB notes use block-friendly form.
- **Test:** `test_BC_2_02_008_edge_cases()` + fuzz `fuzz_store_parse` (VP-016 seed; full target in S-3.03)
