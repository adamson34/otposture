# S-2.02: Catalog Canonicalization & Content Digest

## Narrative
- **As the** comparability guard, **I want** content-bound catalog identity, **so that** a tampered or repackaged catalog claiming the same version is detected, not trusted.

## Acceptance Criteria

### AC-001 (traces to BC-1.01.003 postconditions 1–2)
- `digest(&ValidCatalog) -> ContentDigest` = SHA-256 over canonicalized content; exposed alongside `(catalog_id, version)`; mismatch path yields typed error for E-CAT-004 rendering.
- **Test:** `test_BC_1_01_003_digest_identity()`

### AC-002 (traces to BC-1.01.003 EC-002, VP-005)
- Key order, whitespace, comment, and encoding variations of the same semantic content produce equal digests.
- **Test:** proptest `prop_BC_1_01_003_canonicalization_invariant()`

### AC-003 (traces to BC-1.01.003 VP-006)
- Any semantic change (one title char, one weight, one threshold) changes the digest.
- **Test:** proptest `prop_BC_1_01_003_digest_sensitivity()`

### AC-004 (traces to BC-1.01.003 invariant 1)
- Loaded catalogs are immutable: no `&mut` API exists post-validation.
- **Test:** compile-time (API review) + doc test
