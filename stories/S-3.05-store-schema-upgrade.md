# S-3.05: Store Schema Upgrade

## Narrative
- **As a** long-term user updating the binary, **I want** explicit, backed-up schema migration, **so that** a new version can never strand or mangle my history.

## Acceptance Criteria

### AC-001 (traces to BC-2.02.009 postconditions 1–4)
- `upgrade`: timestamped full backup inside the store before any rewrite (announced); rewrite via atomic discipline; upgraded store loads clean; semantic content preserved exactly (VP-025 — snapshot count, seqs, answers, scores recompute-match, overrides, breaks); idempotent no-op on current version.
- **Test:** `test_BC_2_02_009_upgrade()` + proptest VP-025/VP-027

### AC-002 (traces to BC-2.02.009 invariant 3, EC-002, VP-026)
- Any single injected fault → original-intact (pre-switch-over) or upgrade-complete (post); schema marker flips as the final atomic step; backup present once switch-over begins.
- **Test:** fault-injection `fault_BC_2_02_009_crash_safety()`

### AC-003 (traces to BC-2.02.009 EC-001/003/004)
- ENOSPC during backup → E-FS-002, original untouched, partial backup cleaned; multi-version chains internally with one backup; quarantined orphans carried through unchanged.
- **Test:** `test_BC_2_02_009_edge_cases()`

### AC-004 (traces to BC-2.02.009 invariant 1)
- No other command rewrites store schema on disk (compatibility stays in-memory per BC-2.02.004 EC-003).
- **Test:** integration assertion across the command suite (schema file hash stable under non-upgrade commands)
