# S-1.01: Workspace Scaffolding & CI Gates

## Narrative
- **As a** maintainer, **I want** the 6-crate workspace with all quality gates wired from commit one, **so that** purity, offline, and license invariants are enforced before any logic exists.

## Acceptance Criteria

### AC-001 (traces to BC-6.06.003 invariant 1)
- `cargo deny check` fails if any networking crate (reqwest, hyper, ureq, curl, tokio net features, async-std net) enters the workspace feature graph; ban list committed as `deny.toml`.
- **Test:** `test_BC_6_06_003_deny_bans_networking_crates()` (CI: inject a banned dep in a fixture branch → deny fails)

### AC-002 (traces to NFR-009 / VP-051)
- CI job fails if `op-score`'s dependency tree contains any I/O-capable crate; `#![forbid(unsafe_code)]` set in op-score, op-assess, op-render, op-store::codec module.
- **Test:** `test_purity_lint_op_score_no_io_deps()` (cargo metadata assertion script)

### AC-003 (traces to ADR-001 consequences)
- `cargo metadata` graph matches dependency-graph.md edge list exactly (op-score has zero workspace deps; op-render does not depend on op-assess).
- **Test:** `test_dependency_graph_matches_architecture()`

### AC-004 (traces to tooling-selection.md CI matrix)
- CI workflow runs build+test on the 4-target matrix, plus kani/fuzz/mutants/deny job skeletons (jobs exist and pass trivially on the empty workspace).
- **Test:** CI green on initial PR.
